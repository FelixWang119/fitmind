## 1. 需求概述

构建一个灵活、可扩展的通知系统，支持多种通知渠道、用户自定义设置和智能触发机制，为 FitMind 用户提供及时、有价值的健康提醒和鼓励消息。

### 2.2 核心组件

#### 2.2.1 通知触发机制（轮询方案）

**架构决策：** 采用轮询而非事件系统，原因：
- 系统并发量低，不需要高并发架构
- 分钟级实时性可接受，不需要秒级响应
- 轮询更易于调试和追溯
- 减少系统复杂度和运维成本

| 触发器类型 | 触发方式 | 轮询频率 | 描述 |
|-----------|---------|---------|------|
| **习惯提醒** | 定时轮询 | 每分钟 | 查询待发送的习惯提醒 |
| **里程碑检测** | 定时轮询 | 每 5 分钟 | 查询用户达成的里程碑 |
| **主动关怀** | 定时任务 | 每天 8:00/14:00 | 早安关怀、未登录关怀 |
| **事件日志处理** | 定时轮询 | 每 30 秒 | 处理 pending 状态的事件日志 |

#### 2.2.2 事件日志表（解耦设计）

**目的：** 解耦业务系统和通知系统

```
业务系统 → 写入 event_logs 表 → 通知系统轮询处理
```

**优势：**
- 通知系统不依赖业务表结构
- 新增业务类型无需修改通知系统
- 统一审计和追溯
- 天然支持重试

#### 2.2.3 通知服务（NotificationService）

核心职责：
- 创建通知记录
- 检查用户通知设置
- 通知去重
- 优先级判断
- 渠道路由（Web 应用内 + 邮件）

#### 2.2.3 通知队列（Notification Queue）

**Phase 1 方案：** 数据库队列
- 使用 PostgreSQL 的 `notifications_queue` 表
- 定时任务轮询处理

**Phase 2 方案：** Redis 队列
- 使用 Redis 作为消息队列
- 支持延迟队列、优先级队列
- 更好的并发性能

#### 2.2.4 渠道适配器（Channel Adapters）

采用策略模式，支持多渠道：

```python
class NotificationChannel(ABC):
    @abstractmethod
    async def send(self, notification: Notification) -> bool:
        pass

class InAppChannel(NotificationChannel):
    """App 内通知渠道（Phase 1）"""
    pass

class EmailChannel(NotificationChannel):
    """邮件通知渠道（Phase 2）"""
    pass

class PushChannel(NotificationChannel):
    """推送通知渠道（Phase 2）"""
    pass
```

## 3. 数据模型设计

### 3.1 通知记录表（notifications）

```sql
CREATE TABLE notifications (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    
    -- 通知内容
    notification_type VARCHAR(50) NOT NULL,  -- 'habit_reminder', 'milestone', 'care', 'system'
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    template_id UUID REFERENCES notification_templates(id),  -- 使用的模板
    template_data JSONB,  -- 模板变量数据
    
    -- 通知分类
    category VARCHAR(50) NOT NULL,  -- 'reminder', 'celebration', 'info', 'alert'
    priority INTEGER DEFAULT 5,  -- 1-10，10 最高
    
    -- 渠道信息
    channel VARCHAR(50) NOT NULL,  -- 'in_app', 'email', 'sms', 'push'
    channel_data JSONB,  -- 渠道特定数据（如邮件主题、推送 payload）
    
    -- 状态
    status VARCHAR(20) DEFAULT 'pending',  -- 'pending', 'sent', 'delivered', 'read', 'failed'
    sent_at TIMESTAMP,
    read_at TIMESTAMP,
    delivered_at TIMESTAMP,
    
    -- 错误处理
    error_message TEXT,
    retry_count INTEGER DEFAULT 0,
    max_retries INTEGER DEFAULT 3,
    
    -- 去重控制
    deduplication_key VARCHAR(100),  -- 去重键
    deduplication_window INTEGER DEFAULT 3600,  -- 去重窗口（秒）
    
    -- 元数据
    source_type VARCHAR(50),  -- 触发源类型
    source_id VARCHAR(100),   -- 触发源 ID
    metadata JSONB,
    
    -- 时间戳
    created_at TIMESTAMP DEFAULT NOW(),
    scheduled_at TIMESTAMP,  -- 计划发送时间（用于延迟发送）
    expires_at TIMESTAMP     -- 过期时间
);

-- 索引
CREATE INDEX idx_notifications_user_id ON notifications(user_id);
CREATE INDEX idx_notifications_status ON notifications(status);
CREATE INDEX idx_notifications_user_status ON notifications(user_id, status);
CREATE INDEX idx_notifications_scheduled_at ON notifications(scheduled_at) 
    WHERE status = 'pending';
CREATE INDEX idx_notifications_dedup_key ON notifications(deduplication_key, created_at);
```

```sql
CREATE OR REPLACE FUNCTION create_default_notification_settings()

    VALUES (NEW.id);
    RETURN NEW;

    FOR EACH ROW
```

## 5. 核心服务设计

### 5.1 NotificationService

```python
class NotificationService:
    """通知服务"""
    
    def __init__(self, db: Session, channel_factory: ChannelFactory):
        self.db = db
        self.channel_factory = channel_factory
        self.template_engine = TemplateEngine()
    
    async def create_and_send(
        self,
        user_id: UUID,
        notification_type: str,
        title: str,
        content: str,
        category: str = "info",
        priority: int = 5,
        channel: str = "in_app",
        template_data: dict = None,
        deduplication_key: str = None,
        scheduled_at: datetime = None,
        metadata: dict = None,
    ) -> Notification:
        """创建并发送通知"""
        
        # 1. 检查用户通知设置
        settings = await self._get_user_settings(user_id)
        if not self._should_send_notification(settings, notification_type, channel):
            logger.info(f"Notification skipped for user {user_id}: disabled by settings")
            return None
        
        # 2. 检查免打扰时间
        if self._is_in_do_not_disturb(settings):
            # 延迟到免打扰结束后发送
            scheduled_at = self._get_do_not_disturb_end_time(settings)
        
        # 3. 去重检查
        if deduplication_key:
            if await self._is_duplicate(user_id, deduplication_key, settings):
                logger.info(f"Duplicate notification skipped: {deduplication_key}")
                return None
        
        # 4. 创建通知记录
        notification = Notification(
            user_id=user_id,
            notification_type=notification_type,
            title=title,
            content=content,
            category=category,
            priority=priority,
            channel=channel,
            template_data=template_data,
            deduplication_key=deduplication_key,
            scheduled_at=scheduled_at,
            metadata=metadata,
        )
        self.db.add(notification)
        self.db.commit()
        
        # 5. 加入队列（如果是定时发送）
        if scheduled_at and scheduled_at > datetime.now():
            await self._enqueue_notification(notification, scheduled_at)
        else:
            # 6. 立即发送
            await self._send_notification(notification)
        
        return notification
    
    async def send_habit_reminder(
        self,
        user_id: UUID,
        habit_name: str,
        habit_id: UUID,
    ):
        """发送习惯提醒通知"""
        template = await self._get_template('habit_reminder')
        title = self.template_engine.render(template.title_template, {'habit_name': habit_name})
        content = self.template_engine.render(template.content_template, {'habit_name': habit_name})
        
        await self.create_and_send(
            user_id=user_id,
            notification_type='habit_reminder',
            title=title,
            content=content,
            category='reminder',
            priority=7,
            channel='in_app',
            template_data={'habit_name': habit_name},
            deduplication_key=f'habit_reminder:{habit_id}:{datetime.now().date()}',
            metadata={'habit_id': str(habit_id)},
        )
    
    async def send_milestone_notification(
        self,
        user_id: UUID,
        milestone_type: str,
        milestone_data: dict,
    ):
        """发送里程碑通知"""
        template_code = self._get_milestone_template_code(milestone_type)
        template = await self._get_template(template_code)
        
        title = self.template_engine.render(template.title_template, milestone_data)
        content = self.template_engine.render(template.content_template, milestone_data)
        
        await self.create_and_send(
            user_id=user_id,
            notification_type='milestone',
            title=title,
            content=content,
            category='celebration',
            priority=9,
            channel='in_app',
            template_data=milestone_data,
            deduplication_key=f'milestone:{milestone_type}:{datetime.now().date()}',
            metadata={'milestone_type': milestone_type, **milestone_data},
        )
```

```python
        self.notification_service = notification_service
    
    async def detect_and_notify(self, user_id: UUID):
        """检测里程碑并发送通知"""
        milestones = await self._detect_milestones(user_id)
        
        for milestone in milestones:
            # 创建记忆
            await self._create_memory(user_id, milestone)
            
            # 发送通知
            await self.notification_service.send_milestone_notification(
                user_id=user_id,
                milestone_type=milestone['type'],
                milestone_data=milestone['data'],
            )
```

## 8. 通知价值与打扰平衡设计

**审阅备注：** 本节为架构师 Winston 审阅后添加（2026-02-25）

### 8.1 核心设计原则

| 原则 | 描述 |
|------|------|
| **用户控制优先** | 让用户感觉自己在控制通知，而不是被通知控制 |
| **价值驱动** | 每条通知都应该有明确的用户价值 |
| **数据驱动** | 基于用户行为持续优化通知策略 |
| **适度克制** | 宁可少发，不要过度 |
| **场景感知** | 时间、地点、用户状态都要考虑 |

### 8.2 分层决策架构

### 8.3 通知价值评分模型

**评分范围：** 0-100 分

**决策阈值：**
- **≥ 70 分**：立即发送（高价值通知）
- **50-69 分**：延迟到最佳时间发送（中等价值）
- **< 50 分**：不发送或聚合（低价值）

**评分因子：**

| 因子 | 权重 | 说明 |
|------|------|------|
| 通知类型基础分 | 50% | 不同类型有不同基础价值 |
| 用户历史行为 | 20% | 同类通知打开率 |
| 时间敏感性 | 10% | 是否紧急/时效性 |
| 个性化相关度 | 20% | 与用户当前目标的相关性 |

**FitMind 通知类型基础分建议：**

| 通知类型 | 基础分 | 说明 |
|---------|--------|------|
| 里程碑达成 | 90 | 高价值，用户需要正反馈 |
| 习惯提醒（用户主动设置） | 70 | 用户明确需求 |
| 连续打卡庆祝 | 75 | 游戏化激励 |
| 早安关怀 | 40 | 低价值，易变骚扰 |
| 连续未登录关怀 | 50 | 中等价值 |
| 系统通知 | 30 | 最低优先级 |

### 8.4 疲劳度管理策略

**疲劳度计算窗口：** 7 天滑动窗口

**疲劳等级：**

| 等级 | 分数 | 特征 | 限制措施 |
|------|------|------|---------|
| 低疲劳 | 0-29 | 正常接收通知 | 无限制 |
| 中疲劳 | 30-59 | 打开率下降 | 只发送优先级≥7 的通知 |
| 高疲劳 | 60-100 | 大量关闭通知 | 只发送优先级≥9 的通知 |

**疲劳度计算因子：**
- 过去 7 天通知数量（权重 30%）
- 过去 7 天通知打开率（权重 40%）
- 过去 7 天通知关闭率（权重 30%）

### 8.5 通知渠道策略

**架构决策：** 所有通知默认 Web 应用内，重要通知额外发送邮件

#### 渠道定义

| 渠道 | Phase | 说明 |
|------|-------|------|
| **Web 应用内** | Phase 1 | 通知中心 + Toast 弹窗，60 秒轮询未读数量 |
| **邮件** | Phase 1 | 仅重要通知（里程碑、周报等） |
| **浏览器推送 (Web Push)** | Phase 2 可选 | 需用户授权，支持锁屏通知 |
| **短信** | ❌ 不推荐 | 成本高、体验差 |

#### 通知分级与渠道路由

| 通知类型 | 分级 | 渠道 | 示例 |
|---------|------|------|------|
| 习惯完成 | 普通 | Web 应用内 | 习惯打卡完成 |
| 连续 7 天打卡 | 普通 | Web 应用内 | 小成就 |
| 连续 30 天打卡 | 重要 | Web 应用内 + 邮件 | 重大里程碑 |
| 体重目标达成 | 重要 | Web 应用内 + 邮件 | 核心目标 |
| 徽章解锁 | 普通 | Web 应用内 | 游戏化元素 |
| 周报/月报 | 重要 | Web 应用内 + 邮件 | 阶段性总结 |
| 早安/关怀 | 普通 | Web 应用内 | 日常问候 |

#### 实时通信方案

**架构决策：** 轮询而非 SSE/WebSocket

| 方案 | 决策 | 理由 |
|------|------|------|
| 打开 App 时加载 | ✅ 采用 | 即时获取通知 |
| 下拉刷新 | ✅ 采用 | 用户主动触发 |
| 60 秒轮询未读数量 | ✅ 采用 | 更新红点状态 |
| 关键操作后刷新 | ✅ 采用 | 习惯打卡后刷新 |
| SSE | ❌ 不采用 | 移动端支持差、过度设计 |
| WebSocket | ❌ 不采用 | 复杂度高、不需要秒级实时 |

## 12. 架构决策记录

### 关键决策总结

| 决策点 | 选项 | 决策 | 理由 |
|--------|------|------|------|
| **触发机制** | 事件系统 vs 轮询 | ✅ 轮询 | 简单、易调试、适合低并发 |

| **重要通知定义** | 所有通知 vs 部分通知 | ✅ 部分通知 | 邮件成本考虑、减少打扰 |
