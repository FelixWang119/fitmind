**Created:** 2026-02-21T07:50:58

## Overview

### Problem Statement

肥胖客户需要可负担的、专业的、AI驱动的体重管理支持，提供情感价值和长期陪伴，需要简单易用的开发环境。

### Solution

构建一个Docker化的Web应用，使用React前端、FastAPI后端、PostgreSQL数据库，集成通义千问AI，提供专业营养和行为指导、科学数据跟踪。

### Scope

**In Scope:**
- Docker容器化部署
- Web界面
- 聊天功能
- 用户注册
- 健康数据跟踪
- AI对话
- 习惯建议
- 基础仪表板
- 完整开发环境配置

**Out of Scope:**
- 移动应用
- 支付系统

- 社交网络功能
- 高级监控和运维

### Technical Decisions

#### 核心技术栈
1. **前端：** React + TypeScript + Tailwind CSS
2. **后端：** Python FastAPI + SQLAlchemy
3. **数据库：** PostgreSQL 15
4. **AI服务：** 通义千问(QWen) API + LangChain集成
5. **容器化：** Docker + Docker Compose
6. **开发工具：** Vite (前端构建), Poetry (Python依赖管理)

#### 架构决策记录

**ADR 001: Docker编排选择**
- **状态：** 已接受
- **上下文：** 需要平衡开发便利性和生产就绪性
- **决策：** 使用Docker Compose with生产准备配置
- **后果：** 
  - 正面：简化开发，生产就绪
  - 负面：配置略复杂，需要学习Docker最佳实践
- **实现：** 单一docker-compose.yml文件，通过环境变量区分开发/生产模式
- **团队建议：** 添加健康检查、资源限制、网络隔离，使用`postgres:15-alpine`减少体积

**ADR 002: 数据库设计**
- **状态：** 已接受（单体应用简化）
- **上下文：** 健康数据需要灵活查询，作为单体应用保持简单
- **决策：** PostgreSQL关系模型，保持简单设计
- **后果：** 初始简单，维护容易，未来需要时再扩展
- **实现：** 核心表结构，避免过度设计
- **团队建议：** 使用Alembic进行数据库迁移，暂不配置pgvector等复杂扩展

**ADR 003: AI集成模式**
- **状态：** 已接受（简化风险处理）
- **上下文：** 需要安全、可测试的AI服务集成，内部原型可简化风险处理
- **决策：** API网关模式 + 开发模拟，异步处理简化
- **后果：** 解耦AI服务，便于测试，内部使用风险可控
- **实现：** AI服务层抽象，开发环境模拟器，基础错误处理
- **团队建议：** 前端显示API加载状态，开发环境使用Mock API避免测试费用，生产环境基础错误重试

**ADR 004: 安全与合规**
- **状态：** 已接受（内部原型简化）
- **上下文：** 医疗健康数据需要特别保护，但作为内部原型可简化
- **决策：** 基础安全配置，合规要求简化
- **后果：** 减少初始配置复杂度，但未来生产环境需要加强
- **实现：** 非root容器用户、基础数据保护、开发环境简化安全
- **团队建议：** 内部原型阶段可简化安全要求，未来生产环境需添加完整合规框架

**ADR 005: 开发体验优化**
- **状态：** 已接受
- **上下文：** 用户不熟悉Docker操作
- **决策：** 简化命令包装 + 详细文档
- **后果：** 降低入门门槛，增加维护文档工作
- **实现：** 包装脚本、操作指南、故障排除文档
- **团队建议：** 提供`./dev.sh start`、`./dev.sh test`、`./dev.sh logs`等简化命令

**ADR 006: 用户体验设计**
- **状态：** 已接受
- **上下文：** 体重管理是敏感话题，需要温暖专业的界面
- **决策：** 情感智能设计 + 渐进式交互
- **后果：** 增强用户粘性，可能增加前端复杂度
- **实现：** 柔和的色彩方案，结构化聊天流程，微动画反馈
- **团队建议：** 使用CSS变量实现主题切换，为黑暗模式预留，检测用户类型调整交互方式

**ADR 007: 产品验证策略**
- **状态：** 已接受
- **上下文：** 需要验证核心价值主张
- **决策：** 最小可测试产品 + 用户反馈循环
- **后果：** 快速迭代，避免功能蔓延
- **实现：** 1周内让5个测试用户使用，收集反馈
- **团队建议：** 明确目标用户细分（初次尝试者 vs 多次失败者），设计不同的情感支持方式

### 技术类比（帮助理解）

| 技术概念 | 生活类比 | 简单解释 |

| **PostgreSQL数据库** | 电子表格 | 像Excel存数据，但更强大安全 |
| **FastAPI后端** | 餐厅厨房 | 处理请求，准备数据，服务前端 |
| **React前端** | 餐厅菜单 | 用户看到的界面，点菜交互 |

### Dependencies

#### 外部依赖
- **Docker Desktop** 或 **Docker Engine** (版本20.10+)
- **Node.js** (版本18+，通过Docker容器提供)
- **Python** (版本3.11+，通过Docker容器提供)
- **通义千问API密钥** (生产环境必需，开发环境可选)
- **PostgreSQL 15** (通过Docker容器提供)

#### 服务依赖
1. **前端服务依赖：**
   - 后端API服务运行正常
   - 环境变量配置正确

2. **后端服务依赖：**
   - PostgreSQL数据库连接
   - 通义千问API访问（生产环境）
   - 环境变量配置

3. **数据库服务依赖：**
   - 持久化存储卷
   - 足够的磁盘空间

#### 任务依赖关系
- Task 1-2 → 所有其他任务（基础环境）
- Task 5 → Task 6（认证需要用户表）
- Task 6 → Task 7-9（数据模型）
- Task 8 → Task 7（聊天需要AI服务）
- Task 1-10完成 → 阶段2任务开始

### Testing Strategy

- **前端：** Jest + React Testing Library
  - 组件渲染测试
  - 用户交互测试
  - 状态管理测试
  - 覆盖率目标：70%（核心组件必须测试）

- **后端：** pytest
  - 服务层逻辑测试
  - API端点测试
  - 数据模型测试
  - 覆盖率目标：75%（核心业务逻辑必须测试）

### Notes

1. **Docker性能问题：** 用户担心本地机器性能，需要优化资源限制
2. **AI成本控制：** 通义千问API调用费用需要监控和控制
3. **数据安全合规：** 健康数据保护需要基础合规框架
4. **用户接受度：** AI营养建议的价值需要快速验证

#### 已知限制
1. **初始版本功能有限：** 专注于核心价值验证
2. **移动端体验：** 响应式设计但非原生移动应用
3. **AI准确性：** 依赖第三方AI模型，无法完全控制输出质量
4. **多语言支持：** 仅支持中文初始版本

#### 对抗性审查优化总结
基于对抗性审查反馈和实际情况调整：
1. **安全合规**：内部原型简化要求，专注功能验证
2. **AI集成**：接受基础风险，简化错误处理
3. **性能指标**：调整响应时间要求（AI相关10秒，非AI相关500ms）
4. **测试策略**：通过率85%，覆盖率目标调整（前端70%，后端75%）
5. **架构设计**：确认单体应用，避免复杂扩展（移除pgvector等）
6. **部署策略**：内部原型简化部署要求
7. **技术债务**：接受合理技术债务，快速验证核心价值
8. **依赖管理**：基础版本锁定，简化许可证合规
9. **扩展性**：保持简单设计，未来需要时再扩展
10. **用户体验**：保持简单验证，渐进式优化
