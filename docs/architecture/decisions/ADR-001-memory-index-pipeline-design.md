# 架构决策记录 (ADR)

## ADR-001: 记忆索引管道设计 - SQLite队列+溢出触发 vs 定期索引

### 状态
- **状态**: 已接受
- **日期**: 2026-02-27
- **决策者**: 架构团队
- **相关方**: 后端开发团队、AI服务团队

### 背景
在健康管理AI助手中，需要实现一个记忆系统，将用户的短期行为（餐食、运动、健康记录）转换为长期记忆，以便AI能够提供个性化的建议。

**核心需求**:
1. 实时性：新记录需要尽快进入记忆系统
2. 可靠性：重启后记忆不丢失
3. 性能：不影响主业务流程
4. 可扩展性：支持多种数据类型

**当前问题**:
- 开发人员对记忆系统设计理解不一致
- 存在多个实现版本
- 缺乏明确的架构文档指导

### 考虑的方案

#### 方案1: SQLite队列 + 溢出触发索引 (已接受方案)
- **架构**: 两级记忆系统（SQLite持久化队列 + PostgreSQL长期存储）
- **触发机制**: 当短期记忆队列达到100条阈值时，新记录进入触发最旧记录溢出并立即索引
- **存储**: SQLite作为队列存储，PostgreSQL作为长期存储
- **优点**: 
  - 实时性好，新记录立即进入短期记忆
  - 重启后不丢失（SQLite持久化）
  - 自动溢出保护，无需定期调度
  - 简化架构，移除定期索引管道
- **缺点**: 
  - 需要维护两个存储系统
  - 复杂性较高
  - 需要处理数据同步

#### 方案2: 定期批量索引管道 (已否决方案)
- **架构**: 单一数据库存储，定期扫描新记录
- **触发机制**: 定时任务（如每天凌晨3点运行）
- **存储**: 仅PostgreSQL
- **优点**: 
  - 架构简单
  - 易于理解和维护
  - 数据一致性容易保证
- **缺点**: 
  - 实时性差（最长延迟24小时）
  - 重启后新记录可能丢失
  - 高峰期可能造成数据库压力
  - 需要复杂的调度系统
  - 与SQLite队列设计冲突，造成架构混乱

#### 方案3: 实时同步索引 (已否决方案)
- **架构**: 事务内同步索引
- **触发机制**: 每次创建记录时立即索引
- **存储**: PostgreSQL
- **优点**: 
  - 实时性最好
  - 数据一致性最强
- **缺点**: 
  - 严重影响性能
  - 增加事务复杂度
  - 可能阻塞主业务流程
  - 不适合高频用户场景

### 决策

**选择的方案**: 方案1 - SQLite队列 + 溢出触发索引

**决策理由**:
1. **平衡实时性与性能**: SQLite队列提供即时响应，溢出触发避免定期调度的复杂性
2. **数据可靠性**: SQLite确保重启后数据不丢失，无需启动加载器
3. **系统简化**: 移除定期索引管道，减少架构复杂度
4. **与最新设计一致**: 符合最新的短期记忆架构文档
5. **已验证的有效性**: SQLite队列已满足SLA要求，无需额外安全网

**预期结果**:
- 新记录在1秒内进入短期记忆系统
- 索引过程不影响主业务流程
- 系统重启后记忆数据完整保留
- 支持每天数千条记录的索引需求

### 后果

#### 积极后果
- 提供接近实时的记忆能力
- 系统性能稳定，不影响用户体验
- 架构清晰，易于扩展新数据类型
- 自动负载均衡，适应流量变化

#### 消极后果/风险
- **复杂性风险**: 需要维护两个存储系统
  - *缓解措施*: 封装存储抽象层，提供统一接口
- **数据同步风险**: 短期和长期存储可能不一致
  - *缓解措施*: 实现幂等性索引，定期一致性检查
- **SQLite性能风险**: 高并发下可能成为瓶颈
  - *缓解措施*: 监控队列长度，优化SQLite配置
