# 习惯管理系统重新设计 - 变更日志

**日期**: 2026-03-01  
**版本**: PRD v1.3  
**变更类型**: 重大设计调整

---

## 📋 变更概述

### 废弃的功能
- ❌ **自动关联规则引擎** (Phase 2)
  - 饮食/运动记录自动完成习惯的逻辑
  - 规则引擎服务 (`HabitAutoCompleteService`)
  - 数据库字段 `habits.auto_complete_rule`
  - 数据库字段 `habit_completions.source_type/source_id`

### 新增的功能
- ✅ **5 个固定习惯系统**
  1. 每天保持热量赤字（自动计算，复用饮食 + 运动）
  2. 每天 8 杯水（独立饮水页面，点亮 8 个水杯）
  3. 每天记录体重（独立体重页面，支持多次记录）
  4. 每日早睡（独立早睡页面，情绪价值鸡汤）
  5. 每日正念冥想（独立冥想页面，灵性语录）

- ✅ **Dashboard 新布局**
  - 第 0 行：欢迎 Banner
  - 第 1 行：热量净变化（核心 Card）
  - 第 2 行：运动打卡 + 饮食打卡
  - 第 3 行：4 个习惯快捷入口（饮水/体重/早睡/冥想）
  - 第 4 行：AI 健康顾问

- ✅ **习惯管理页重新定位**
  - 只做习惯创建、汇总、分析
  - 不直接打卡（打卡在独立页面或 Dashboard）
  - 习惯开通引导流程（价值说明）

---

## 🎯 设计原则变更

### 变更前
```
习惯打卡 = 可以自动完成
         = 与饮食/运动数据关联
         = 计数累计型 + 二元完成型混合
```

### 变更后
```
习惯打卡 = 独立手动完成
         = 与饮食/运动数据隔离
         = 只支持二元完成型（是/否）
         
计数累计型需求 → 独立模块（如饮水页面）
```

## 🎓 决策记录

### 为什么废弃自动关联引擎？

**原因**:
1. **复杂度高**：需要规则引擎、事件触发、冲突处理（2-3 天开发）
2. **用户困惑**："为什么我记录饮食就算完成习惯？"
3. **功能冗余**：Dashboard 已有热量 Card，无需重复
4. **定位不清**：习惯打卡是"记录数据"还是"追踪坚持"？

**新方案优势**:
1. **架构清晰**：饮食/运动=数据层，习惯=追踪层，职责分明
2. **用户体验**：Dashboard 一键打卡，路径最短
3. **开发成本低**：独立页面，无复杂关联逻辑
4. **可扩展**：未来可随时添加更多习惯
